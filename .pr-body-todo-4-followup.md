## Summary

Follow-up to #19. The shipped `Sync GHCR package visibility` workflow step was misleading on user-owned packages: it logged `package not registered yet (first run?); skipping` when the real cause was a hard GitHub-API limitation — `PATCH /user/packages/.../visibility` requires a PAT with `write:packages` scope, which the workflow's default `GITHUB_TOKEN` doesn't carry. This PR replaces the misleading log with a structured `::notice::` that explains exactly what to do.

## Changes

- `template/.github/workflows/build-and-push.yml` — rewrites the visibility-sync step:
  - Logs a `context:` line (`user-owned (default GITHUB_TOKEN — expected to 404; manual one-time link required)` vs `org-owned (auto-sync via default GITHUB_TOKEN)`) so the run trace is self-explanatory.
  - On user-owned 404, emits `::notice title=GHCR visibility::...` with the exact UI procedure (Profile → Packages → Package settings → Manage Actions access → connect repo → Change visibility).
  - Adds opt-in `secrets.GHCR_PAT` support: `GH_TOKEN: ${{ secrets.GHCR_PAT || secrets.GITHUB_TOKEN }}`. If the user supplies a PAT with `write:packages` scope, user-owned visibility auto-syncs cleanly.
- `template/NEXT_STEPS.md.j2` — replaces the "auto-sync works universally" framing in the "Container image (GHCR)" subsection with an explicit user-vs-org split, both workarounds (one-time UI link + GHCR_PAT), and a multi-arch note for Apple Silicon contributors.

## Why this matters

Without this fix, every user-owned repo scaffolded from pbg-template's `main` will see a misleading workflow log on its first push and silently leave the GHCR package private. The honesty fix surfaces the limitation immediately with actionable next steps.
