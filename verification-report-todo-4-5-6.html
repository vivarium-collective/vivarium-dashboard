#4 — GHCR package visibility PARTIAL

Goal: anonymous docker pull against the published image works "just" after the first workflow run, without anyone having to flip a setting manually.

Hard GitHub-API limitation discovered during verification. The workflow's default GITHUB_TOKEN cannot PATCH user-owned package visibility — that endpoint requires a Personal Access Token with write:packages scope, which workflow tokens don't carry. Org-owned packages auto-sync fine; user-owned packages stay private until either (a) the user adds a GHCR_PAT secret or (b) does a one-time UI link. The currently-shipped workflow logs a misleading "package not registered yet" line on user-owned 404s; the honesty fix (Chunk B of this session, becoming a follow-up PR) replaces that with a structured ::notice:: with explicit one-time-UI instructions.
  • User-owned: clear messaging instead of misleading log. Chunk B's revised workflow emits a ::notice title=GHCR visibility:: with the exact one-time-UI procedure on a user-owned 404. Verified by code review of the rewritten step in ../pbg-template/template/.github/workflows/build-and-push.yml on the local branch.
  • GHCR_PAT opt-in. Workflow now reads ${{ secrets.GHCR_PAT || secrets.GITHUB_TOKEN }}; with a user-supplied PAT, user-owned visibility flips cleanly. Verified by code review.
  • Docs honest about user-vs-org. template/NEXT_STEPS.md.j2's "Container image (GHCR)" subsection documents the split and both workarounds (one-time UI link + GHCR_PAT secret).
  • Org-owned auto-sync via GITHUB_TOKEN. Workflow code path uses PATCH /orgs/.../visibility when github.repository_owner_type == "Organization". Untested against a live org repo this session — no vivarium-collective/<demo> workspace was scaffolded. Deferred until an org-owned demo repo is needed.
  • Same recipe with --private keeps the image private. Deferred.

syncing ghcr.io/alexpatrie/hpc-demo-ws visibility to 'public' via /user/packages/container/hpc-demo-ws/visibility
::notice title=GHCR visibility::User-owned packages cannot have their visibility flipped by the workflow's default GITHUB_TOKEN (GitHub API limitation). Image is currently PRIVATE. To make anonymous `docker pull` work, do this once: visit https://github.com/AlexPatrie?tab=packages&repo_name=hpc-demo-ws, open the 'hpc-demo-ws' package, click 'Package settings' → 'Manage Actions access' → connect this repository, then under 'Danger Zone' → 'Change package visibility' set to 'public'. After that one-time link, visibility inherits from the repo automatically. Alternative: add a `GHCR_PAT` secret with `write:packages` scope to fully automate this step.
  1. Go to github.com/<owner>?tab=packages.
  2. Click the package (e.g. hpc-demo-ws).
  3. Package settings → Manage Actions access → Add Repository → select your workspace repo → "Role: Write".
  4. Scroll to Danger Zone → Change package visibility → Public.
  5. Done — every future push inherits visibility from the repo automatically.
  1. Create a Personal Access Token (classic) at github.com/settings/tokens/new with write:packages scope only.
  2. In your workspace repo: Settings → Secrets and variables → Actions → New repository secret → name GHCR_PAT → paste the token.
  3. Next push: the workflow auto-detects the secret and uses it instead of GITHUB_TOKEN for the visibility step. Full automation.
